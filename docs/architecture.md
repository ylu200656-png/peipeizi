# 项目架构说明

## 1. 架构概览

项目采用前后端分离架构：

- 前端负责页面展示、交互逻辑、状态管理和接口调用
- 后端负责业务处理、权限控制、数据存储和对外接口

推荐架构流程：

```text
用户 -> 前端应用 -> HTTP/HTTPS API -> 后端服务 -> 数据库/缓存/文件存储
```

## 2. 模块划分

### 前端模块

- 页面模块：承载业务页面和路由
- 组件模块：沉淀复用 UI 组件
- 服务模块：统一管理 API 请求
- 状态模块：管理全局状态和缓存
- 权限模块：处理登录、鉴权和访问控制

### 后端模块

- 控制器层：接收请求并返回响应
- 服务层：处理核心业务逻辑
- 数据访问层：操作数据库
- 中间件层：处理鉴权、日志、异常、跨域等
- 基础设施层：配置、缓存、文件、消息队列等

## 3. 前后端协作约定

### 接口规范

- 接口路径统一使用 `/api` 前缀
- 接口版本建议使用 `/api/v1`
- 使用 JSON 作为主要数据交换格式
- 时间字段建议统一使用 ISO 8601 或时间戳

### 返回结构建议

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

字段说明：

- `code`：业务状态码，`0` 表示成功
- `message`：返回消息
- `data`：实际业务数据

### 错误处理建议

- 4xx：请求参数、鉴权、权限类错误
- 5xx：服务内部错误
- 前端统一拦截并展示可理解的错误信息

## 4. 环境规划

建议至少区分以下环境：

- `local`：本地开发环境
- `dev`：开发联调环境
- `test`：测试环境
- `prod`：生产环境

## 5. 非功能要求

- 安全性：鉴权、权限控制、敏感信息保护
- 可维护性：分层清晰、命名统一、文档同步
- 可扩展性：模块解耦、配置外置
- 可观测性：日志、监控、告警
