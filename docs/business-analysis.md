# “药捷”药品管理系统业务分析

## 1. 项目定位

根据开题报告，你的项目不是一个面向大型医院的复杂 HIS/ERP 系统，而是一个更偏向于：

- 中小型药店
- 社区卫生服务中心
- 基层卫生机构

的轻量化药品管理系统。

系统目标很明确：

- 替代人工记账和半自动化办公
- 实现药品进、销、存数字化管理
- 做好库存预警和效期预警
- 对国家管制药品进行更严格的监管
- 让普通员工也能快速上手

从这个定位来看，你的系统核心不是“电商平台”，也不是“在线看病系统”，而是一个典型的业务管理系统，重点在于流程规范、数据追溯、库存安全和权限控制。

## 3. 你的系统本质上在解决什么业务问题

你的系统想解决的是药品管理中的 5 个核心问题：

### 1. 药品信息分散

传统模式下，药品名称、规格、批次、供应商、有效期、库存数量等信息分散，查询效率低。

系统要做的事情：

- 建立统一药品档案
- 支持按名称、类别、批次、供应商查询
- 支持药品信息维护

### 2. 入库和销售流程不规范

人工记录容易出现漏记、错记、账实不一致。

系统要做的事情：

- 规范采购入库流程
- 规范销售出库流程
- 记录每次库存变化
- 保证库存可追溯

### 3. 库存风险无法提前发现

库存过低、药品临期、药品过期如果不能提前发现，会直接影响经营和安全。

系统要做的事情：

- 建立库存预警
- 建立有效期预警
- 做“红线提醒”

### 4. 管制药品监管不严

普通药品和管制药品不能用同一套宽松逻辑处理。

系统要做的事情：

- 对管制药品单独标记
- 限制查询、出库、审批权限
- 加强操作日志和追踪能力

### 5. 管理者缺少经营数据

如果只有流水，没有统计分析，就无法判断哪些药卖得快、库存是否合理。

系统要做的事情：

- 展示销售趋势
- 展示库存波动
- 辅助经营决策

## 4. 系统核心业务模块

结合你的开题报告，系统至少应包含以下业务模块。

### 4.1 用户与权限模块

作用：

- 管理系统用户
- 区分不同岗位权限
- 保护敏感数据

建议角色：

- 系统管理员
- 药房管理员
- 库存管理员
- 销售人员

如果你想更贴合业务，还可以单独加：

- 审核人员

核心逻辑：

- 用户登录
- 角色分配
- 菜单权限控制
- 接口权限控制
- 管制药品相关操作的特殊权限控制

### 4.2 药品基础信息模块

作用：

- 维护药品主数据

建议字段：

- 药品编号
- 药品名称
- 规格
- 单位
- 分类
- 生产厂家
- 供应商
- 是否管制药品
- 安全库存
- 保质期

核心逻辑：

- 新增药品
- 修改药品
- 查看药品详情
- 药品分类管理

### 4.3 采购入库模块

作用：

- 记录药品从供应商进入库存的过程

核心逻辑：

1. 创建采购或入库记录
2. 填写药品、批次、数量、进价、供应商、生产日期、有效期
3. 入库成功后增加库存
4. 生成库存流水
5. 支持按批次追溯

这部分很关键，因为后续的销售和效期管理都依赖入库时记录的批次信息。

### 4.4 销售出库模块

作用：

- 记录药品销售或出库行为

核心逻辑：

1. 查询可销售库存
2. 选择药品批次
3. 校验库存是否充足
4. 校验是否过期
5. 如果是管制药品，执行更严格校验
6. 出库成功后扣减库存
7. 生成销售记录和库存流水

这里的重点不是“卖出去”本身，而是“扣库存要准确、可追溯、不能卖过期药”。

### 4.5 库存管理模块

作用：

- 实时反映当前库存情况

核心逻辑：

- 查询当前库存
- 按药品、批次、分类筛选
- 库存盘点
- 库存调整
- 展示库存上下限状态

这部分是系统的核心中台，因为入库、销售、预警都围绕库存展开。

### 4.6 预警模块

作用：

- 提前发现风险

建议至少包含两类预警：

- 库存不足预警
- 有效期临近预警

核心逻辑：

- 当库存数量低于安全库存时触发预警
- 当某批次药品距离过期日小于阈值时触发预警
- 支持首页红色提醒或消息提醒

这是你的项目特色之一，论文里也很适合重点写。

### 4.7 管制药品监控模块

作用：

- 对特殊药品做更严格的管理

核心逻辑：

- 药品档案中单独标记是否为管制药品
- 管制药品出入库必须记录更完整的信息
- 限制操作角色
- 强化日志留痕
- 必要时可以加入双人确认或审批

如果时间有限，你可以先实现“特殊标记 + 权限限制 + 单独查询 + 单独日志”这一版，已经足够支撑毕业设计。

### 4.8 统计分析模块

作用：

- 给管理者提供直观数据支持

建议内容：

- 药品销售排行
- 近期销售趋势
- 库存分布
- 临期药品数量统计
- 低库存药品数量统计

这部分比较适合前端用图表展示，也是毕业设计答辩时很容易出效果的模块。

## 5. 核心业务流程

### 流程一：药品入库

业务过程：

1. 管理员选择供应商
2. 录入药品信息或选择已有药品
3. 填写批次号、数量、采购价、生产日期、有效期
4. 系统保存入库单
5. 系统增加库存
6. 系统生成库存流水
7. 系统刷新预警状态

业务重点：

- 批次信息必须记录
- 有效期必须记录
- 库存变化必须可追踪

### 流程二：药品销售出库

业务过程：

1. 销售人员搜索药品
2. 系统返回可售库存
3. 选择出库数量和对应批次
4. 系统校验库存、效期、权限
5. 出库成功后扣减库存
6. 生成销售记录
7. 刷新库存与预警

业务重点：

- 不能超库存销售
- 不能销售过期药品
- 管制药品不能由无权限人员处理

### 流程三：库存预警

业务过程：

1. 系统定时或实时检查库存
2. 判断是否低于安全库存
3. 判断是否临近过期
4. 生成预警记录
5. 前端首页展示提醒

业务重点：

- 预警规则明确
- 预警展示要直观

### 流程四：管制药品监管

业务过程：

1. 药品建档时标记为管制药品
2. 系统限制其访问和操作权限
3. 每次出入库都记录详细日志
4. 管理员可以单独查看管制药品流向

业务重点：

- 身份校验
- 操作留痕
- 流向可追溯

## 7. 最核心的业务规则

这是你后面写代码时最需要抓住的部分。

### 库存规则

- 入库会增加库存
- 销售会减少库存
- 库存不能减成负数
- 库存最好按“药品 + 批次”维度管理

### 效期规则

- 每批药品都必须有有效期
- 已过期药品禁止销售
- 临近过期药品触发预警

### 权限规则

- 不同角色看到的菜单和操作不同
- 管制药品操作权限高于普通药品
- 关键操作要保留日志

### 追溯规则

- 每次入库、出库都要形成记录
- 每次库存变化都能追溯来源和去向

## 9. 业务上容易出错的地方

这些地方你后面设计数据库和接口时要特别注意：

- 库存到底按药品总量管，还是按批次管
- 销售时优先扣哪个批次
- 临期预警阈值是 7 天、15 天还是 30 天
- 管制药品是否需要单独审批
- 库存盘点差异如何修正
- 删除业务单据后是否允许回滚库存

从你的题目来看，我建议：

- 库存按批次管理
- 销售优先出最早到期批次
- 临期预警阈值先设为 30 天
- 管制药品先做高权限控制，不一定强制审批

这样开发难度适中，也更容易做完整。

## 10. 结论

你的开题报告对应的系统，本质上是一个：

基于 `Spring Boot + MySQL + Redis + Spring Security + Vue 3 + Element Plus` 的前后端分离药品管理系统。

它的核心业务主线是：

药品建档 -> 采购入库 -> 库存维护 -> 销售出库 -> 预警提醒 -> 管制药品监控 -> 数据统计分析

其中最关键的业务能力是：

- 库存准确
- 批次可追溯
- 效期可预警
- 管制药品可监管
- 权限边界清晰

这套业务逻辑是成立的，而且非常适合作为本科毕业设计来实现。
