# 全栈工程师 AI 协同提问标准工作流

在使用 AI（如 Cursor 或带 Agent 的 IDE）开发全栈项目时，请严格按照以下三个步骤向 AI 发送指令，以建立全局视野并保证代码质量。

## 第一步：全链路拓扑扫描与架构反演

**操作前提**：开启全局上下文（在对话框输入 `@Codebase` 或选中整个工作区）。

**发送以下提示词：**

> 你现在是一位资深的全栈架构师。我已经将整个前后端代码库作为上下文发给了你。请你先不要修改任何代码，只做仔细的阅读和审查。请用专业的中文为我总结这个全栈项目的核心架构信息：
> 1. **全栈技术图谱**：前端、后端、数据库、缓存以及部署/构建工具分别使用了什么核心技术栈？
> 2. **工程拓扑结构**：请梳理并解释目前的目录结构，特别是前后端分离的边界、公共包（如共享的类型定义或工具类）的存放位置。
> 3. **核心数据链路**：描述一个典型的数据请求从前端路由触发，经过网关/控制器，到业务逻辑层，再到数据库交互，最后响应返回的完整生命周期。

---

## 第二步：深挖跨端规约与接口契约（极其重要）

**操作前提**：等待第一步 AI 回答完毕后，紧接着发送以下指令。这能确保 AI 后续生成的代码符合你现有的工程潜规则。

**发送以下提示词：**

> 非常好。现在请深入到具体的业务代码层面，帮我总结一下目前项目中的全栈编码约定：
> 1. **接口契约与通信**：前后端统一的 API 响应格式是什么（如统一的结构体包含 code, message, data）？前端是否有统一的 axios 拦截器？后端是如何做全局异常拦截和处理的？
> 2. **认证与鉴权链路**：用户的登录态（Token/Session）是如何在前端存储、在请求头中传递，并在后端进行中间件拦截和权限校验的？
> 3. **数据持久化规范**：后端是如何与数据库交互的（使用了什么 ORM 框架）？是否有特定的事务处理规范或软删除约定？
> 4. **潜在风险暴露**：在当前的跨端交互或架构设计中，有没有我需要特别注意的特殊逻辑、安全隐患或技术债？

---

## 第三步：沉淀“三维”架构资产（生成记忆文档）

**操作前提**：AI 已经完美理解了项目骨架和潜规则。此时需要将其记忆固化。

**发送以下提示词：**

> 基于你刚才的审查结果，请帮我生成一份详尽的 `FULLSTACK_ARCHITECTURE.md` 文档。这份文档的目的是：以后当我开启新的 AI 会话时，只要让 AI 读取这个文档，就能立刻了解全栈项目的全局规范和跨端契约。
> 请排版清晰，必须包含：技术栈概览、前后端目录说明、核心 API 交互契约、数据库规范，以及新增一个完整全栈功能（从建表、写接口到前端联调）的标准 SOP 流程。

---

## 💡 效率提升指南

拿到生成的 `FULLSTACK_ARCHITECTURE.md` 后，把它保存在项目根目录下的 `doc/` 文件夹里。以后每次想开发新功能，开启新的 AI 对话窗口时，第一句话请务必这样写：

> “请先阅读 `@doc/FULLSTACK_ARCHITECTURE.md`，然后严格按照其中的规范，帮我实现 [填写你的需求] 功能的前后端全链路代码。”
